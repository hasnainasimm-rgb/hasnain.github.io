I Broke Production — Here's What It Taught Me

A post-mortem on a real outage, what triggered it, and the three things I'll never skip again. This isn't a humble-brag. This is what actually happened.

It was a Tuesday afternoon. We were deploying what should have been a routine change — a small update to an IBM ACE message flow that added a new field mapping to an outbound REST call. Twenty minutes after the deployment, the alerts started.

Thousands of messages silently dropping. No error logs. No obvious failure. Just... nothing going through.

What actually happened

The root cause took three hours to find. Here's the timeline:

T+0 — Deploy
New message flow version deployed to integration node. Health check passed. Deployment marked successful.
T+4 min — First sign
Queue depth on INBOUND.REQUEST.Q starts climbing. Nobody notices yet — it's within normal variance.
T+18 min — Alerts fire
Downstream system reports no received messages for 15 minutes. Monitoring threshold crossed. Pagerduty wakes everyone up.
T+23 min — Rollback attempt
We roll back the ACE flow. Queue continues building. It's not the flow version anymore — the message has already mutated something else.
T+2h 40min — Root cause
Discovered that the new field we added had a namespace conflict with a shared schema. Messages were serializing fine but deserializing on the consumer end to an unexpected shape.
The actual bug

We added a field transactionId to the output message. The consuming system already had a field with the same name in a different schema context. No mapping error. No exception. The consumer just silently used its version of the field and dropped the payload as unprocessable.

The three things I'll never skip again

01
Consumer-side schema testing is not optional
We validated our output against our own schema. We didn't validate it against the consumer's expected input schema. These are not the same test. Run integration tests that actually deserialize the message on the other end.
02
Silent drops are worse than loud failures
An exception is easy to catch. A message that is accepted, processed, and silently discarded is invisible until the downstream system reports missing data. Add explicit contract checks on both sides — don't rely on exceptions alone.
03
Rollback doesn't undo side effects
We rolled back the ACE flow thinking it would fix the issue. But rolling back the producer doesn't help if messages are already in the queue in the new format. You need a queue drain strategy, not just a code rollback.

What a proper pre-deployment checklist now looks like

After this incident, we built a checklist that every integration change goes through before hitting production. It's not glamorous. It's a checklist. But it's saved us at least twice since.

  • Does the output message shape match what the consumer expects — not just what we produce?
  • Are there any shared field names with existing schemas? Have they been compared explicitly?
  • What happens to messages already in the queue if we roll back?
  • Is there a test that proves messages arrive and are processed — not just sent?
  • What's the blast radius if this breaks at peak load vs. off-hours?

The best post-mortem outcome isn't "we found the bug." It's "we changed the system so this class of bug can't go undetected again."

The part nobody talks about

Breaking production feels bad. There's a moment in every outage where you're staring at logs, re-reading the same lines, and that hollow feeling sets in — did I do this?

You probably did. Own it cleanly. Write the post-mortem. Share it with the team without burying the human part. Then build the guard that makes it harder to do again.

That's the whole job.

Takeaway

The incident was caused by a namespace collision that was invisible to our test suite because we only tested producer-side validation. The fix was a consumer contract test and a queue drain procedure in the runbook. Total recovery time: 4h 12min. Time to write the post-mortem: embarrassingly long.
